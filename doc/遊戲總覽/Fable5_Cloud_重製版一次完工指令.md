# Fable5 Cloud 重製版一次完工指令

以下內容可直接作為一次性的 `GOAL` 指令，交給高成本模型在 Cloud 環境中自動執行。

這份指令的核心前提只有一個：
**你要它做的是重製版，不是留在舊專案裡繼續修改。**

---

```text
GOAL:
你現在要做的不是在現有專案上繼續修修補補，也不是延續目前 repo 直接往下改。
你要做的是一個**重製版**：在新的位置建立新的交付版本，重新整理、重新收斂、重新落地，但全面參考舊專案已存在的設計文件、程式碼、資產與世界觀。

這裡最重要的定義如下:
1. 這次工作是 **remake / rebuild from references**。
2. 不是在原本專案資料夾上原地續改。
3. 不是在舊 repo 裡繼續疊 patch。
4. 不是把舊 codebase 當成唯一施工地點。
5. 舊專案的用途是：**參考、拆解、抽取、轉用、複製可用內容**。
6. 新專案的用途是：**交付真正完整、可玩、可通關的新版本**。

你必須先做的一件事:
- 在新的工作位置建立這次重製版的專案根目錄。
- 這個新位置必須與舊專案分離。
- 你可以把舊專案視為 reference repo / source archive / asset library。
- 除非只是讀取、參考、複製素材或搬運必要模組，否則不要在舊專案內施工。

你的目標不是 prototype，不是 demo，不是概念片段。
你的目標是一次完成一個「可完整遊玩、可進行通關、可交付」的重製版。

總原則:
1. 新專案交付，舊專案只供參考。
2. 最大化吸收舊專案的設計、素材、邏輯與已驗證方向。
3. 最小化舊專案的混亂、歷史包袱、半完成結構對新版本的拖累。
4. 能直接沿用的 asset 就沿用到極致。
5. 能提煉後轉用的程式碼就提煉後轉用。
6. 不要被迫維持舊專案的檔案結構、耦合與施工痕跡。
7. 要以「新版本能交付」為最高優先，不是以「儘量少動舊 repo」為目標。

任務定義:
你要根據舊專案內容，打造一個新的完整可玩版本，要求如下:

- 有自己的新專案目錄
- 有清楚的新入口與新啟動方式
- 可從開始進入遊戲
- 可理解遊戲目標
- 可承接既有世界觀與角色系統
- 可進入正式遊玩流程
- 可進行實際戰鬥、移動、成長、配置或資源循環
- 可推進至少一條完整世界線、戰役線或正式模式
- 可遇到中後段壓力與 Boss
- 可完成明確通關或結局
- 可看到完整結算
- 可再次開局或回到主流程
- 大量沿用舊專案的角色、視覺、世界、敵人、音樂、UI 或資料成果
- 整體必須看起來像「同一款遊戲的重製版」，而不是陌生新作

你必須先讀的舊專案文件:
- `doc/文件索引.md`
- `doc/遊戲總覽/遊戲設計文件.md`
- `doc/遊戲總覽/開發進度清單.md`
- `doc/系統機制/機制指南.md`
- `doc/介面流程/玩家介面_UserFlow方案.md`
- `doc/介面流程/玩家介面_主畫面與管理介面補充規格.md`
- `doc/介面流程/玩家介面_按鈕與操作總清單.md`
- `doc/介面流程/玩家介面_戰鬥HUD與操作骨架.md`
- `doc/介面流程/主流程與頁面骨架_統一版.md`
- `doc/世界觀/世界觀與視覺圖鑑.md`
- `doc/世界觀/世界故事_輪迴敘事與四隊長人物設計.md`
- `doc/角色與敵方/角色圖鑑.md`
- `doc/角色與敵方/成員圖鑑.md`
- `doc/角色與敵方/怪物圖鑑.md`
- `doc/素材與提示詞/提示詞指南.md`
- `doc/素材與提示詞/隊長立繪_共通Prompt與四星提示詞.md`
- `doc/素材與提示詞/音樂與音效設計總表.md`
- `assets/README.md`
- `progress.md`

你必須盤點並盡量重用的舊專案資產:
- `assets/images/`
- `assets/video/`
- `assets/audio/`
- `assets/transparent-portraits/`
- `assets/images/maps/`
- `assets/images/characters/`
- `assets/images/enemies/`
- `assets/images/props/`

你必須盤點並視情況抽取的舊專案程式碼:
- `src/main.ts`
- `src/ui/`
- `src/hud/`
- `src/world/`
- `src/combat/`
- `src/enemies/`
- `src/captain/`
- `src/progression/`
- `src/economy/`
- `src/skills/`
- `src/squad/`
- `src/totem/`
- `src/data/`
- `src/boss/`

第一階段：審計舊專案，但不要被舊專案綁死
- 先用文件理解遊戲願景、玩法主軸、世界結構、角色定位、通關形式、介面流程
- 再用程式碼找出哪些系統成熟、哪些可抽取、哪些可重用、哪些只是歷史負擔
- 再用素材盤點哪些圖、音樂、影片、UI 元件、頭像、地板、敵人、設施可直接移植
- 你的目標不是維修舊專案，而是判斷哪些東西值得被新重製版吸收

第二階段：建立新專案
- 在新位置建立重製版專案
- 新專案可有自己的檔案結構、自己的入口、自己的流程組織
- 只要有利於完成度，你可以重新整理架構
- 但你必須保留這款遊戲的身份、世界觀與核心設計語言

第三階段：提煉、搬運、重組
- 舊專案中能直接重用的 asset，複製或轉用到新專案
- 舊專案中能抽取的資料、設定、圖像、音樂、影片、角色資料、敵人資料，盡量轉移
- 舊專案中能安全重用的邏輯模組，可抽離後整合進新專案
- 舊專案中若存在大量耦合、髒接線、debug 導向結構，應避免原封不動繼承
- 你的方向是「吸收價值」，不是「繼承混亂」

第四階段：完成正式可通關閉環
你不能停在 demo。你必須做出至少一條完整可通關路線，至少包含:
- 主入口
- 開局
- 角色/隊伍承接
- 正式戰場
- 成長或配置回饋
- 中段推進
- Boss 或等價最終挑戰
- 勝敗條件
- 結算
- 重玩或返回主流程

強制規則:
1. 不要在舊專案上原地繼續蓋。
2. 新版本必須是獨立交付物。
3. 舊專案只可作為參考來源與資產來源。
4. 若要沿用舊 code，應以抽取、移植、重組為主，不是直接在舊位置繼續補。
5. 若要沿用舊 asset，應優先直接使用，而不是忽略後全部重生成。
6. 若舊系統能提供價值，就轉化成新專案可用結構。
7. 若舊系統只是阻礙，就不要為了情懷強行保留。
8. 不准停在占位頁、假流程、示意版 UI。
9. 不准把 debug 工具當正式主流程。
10. 不准交付一個仍需大量人工接手才能真正玩的版本。

決策優先級:
1. 新重製版可完整通關
2. 新重製版可穩定執行
3. 最大化利用舊專案有價值成果
4. 維持原遊戲身份與世界觀
5. 核心玩法清楚
6. 系統深度
7. 額外華麗功能

如果舊文件與舊程式碼互相衝突:
- 優先選擇最接近可交付版本的解法
- 但不要背離原本遊戲的精神與識別性
- 你是在做重製版，所以你可以收斂、重組、刪繁就簡
- 你不是舊架構的奴隸

如果內容規模太大:
- 不准退化成 prototype
- 必須收斂成一條完整正式路線
- 寧可縮範圍，也要保住完整通關閉環
- 其他世界或模式可作為後續延展空間，但本次交付版本必須完整

實作要求:
- 新專案中修掉會阻止完整遊玩的斷點
- 新專案中建立真正的主流程
- 讓角色、戰鬥、敵人、Boss、成長、掉落、結算形成真實互動鏈
- 優先重用舊專案最成熟的素材與設計成果
- 只在必要時新做最小量內容
- 必要時可以重構，但那是為了交付新版本，不是為了清理舊 repo

Git 要求:
- 在新專案中使用 git 管理你的重製版工作
- 每完成一個大里程碑就 commit
- 如環境允許就 push
- 最終要清楚報告新專案位置、commit 結果與交付方式

里程碑建議:
- `audit: extract reusable design, assets, and systems from legacy project`
- `chore: scaffold standalone remake project`
- `feat: build shippable main flow for remake edition`
- `feat: integrate reused art, audio, combat, and progression into remake`
- `feat: add boss clear route and final result loop`
- `fix: close blockers preventing full completion in remake build`

最終交付標準:
你不能把成果描述成「舊專案的新 patch」。
你也不能把成果描述成「原 repo 上繼續改到比較能玩」。
你必須交付的是一個**獨立的新重製版**，只是它有大量吸收舊專案的價值。

最終你必須輸出:
1. 新重製版專案的位置
2. 新重製版實際完成的完整遊玩路線
3. 玩家如何從開始一路玩到通關
4. 你吸收了哪些舊文件決策
5. 你搬運或重用了哪些舊程式碼模組
6. 你搬運或重用了哪些舊 asset
7. 你新增了哪些必要內容
8. 還有哪些缺點存在，但為什麼它仍然算是可交付成品
9. 如何啟動、驗收、重玩
10. 所有 commit 與 push 結果

最後提醒:
- 你做的是 remake，不是 patch。
- 你做的是新交付，不是舊 repo 續修。
- 你要讀懂舊專案，但不要被舊專案困住。
- 舊專案是燃料，不是牢籠。
- 能搬的搬，能用的用，該重組的重組。
- 最後交出來的東西，必須是一個真的能玩到結束的重製版。
```
