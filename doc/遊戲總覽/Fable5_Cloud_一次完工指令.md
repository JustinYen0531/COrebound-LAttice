# Fable5 Cloud 一次完工指令

以下內容可直接作為一次性的 `GOAL` 指令，交給高成本模型在 Cloud 環境中自動執行。

---

```text
GOAL:
你現在不是要做一個小樣、概念驗證或半成品，而是要在這個專案基礎上，盡你所能一次完成一個「可完整遊玩、可進行通關、可交付」的版本。

這不是從零開始，也不是自由發揮的新專案。
你必須把目前 repo 內所有已存在的設計文件、程式碼、素材、音樂、影片、角色設定、世界設定、HUD、流程骨架、資料結構，全部當作可重用資產池，最大化吸收、最大化沿用、最大化落地。你不是來重寫一個陌生遊戲，你是來把這個專案壓榨到極致，重組成真正能交付的一版。

最重要原則:
1. 你的目標不是 prototype，而是「第一個可以完整通關的成品版」。
2. 你必須先全面理解現有 repo，再進入大規模整合與完工。
3. 現有 asset、現有 code、現有 docs 只要能用，就必須優先使用到底。
4. 不准因為偷快而把既有成果丟掉重做一套新的。
5. 不准停在骨架、示意、假的流程、占位頁、僅展示 UI。
6. 必須把核心循環、成長、戰鬥、進場、推進、Boss、結算、再次開局，全部盡量接成一條真的能玩到結束的路線。
7. 除非遇到硬性阻塞，否則不要等待人工介入，你必須自行連續工作到交付版本出現。
8. 每完成一個大里程碑就 commit；如環境允許就 push。
9. 你的標準不是「寫得很漂亮」，而是「真的能玩、真的能通、真的像一款完成度足夠高的遊戲」。

任務定義:
你要做的是基於這個 repo，重製並整合出一個完整可玩的版本，要求如下:

- 可從主入口開始進入遊戲
- 可理解目前遊戲目標
- 可選擇或承接既有角色/隊長/小隊系統
- 可進入正式遊玩流程
- 可進行實際戰鬥、移動、資源取得、成長或配置
- 可推進至少一條完整世界/關卡/挑戰流程
- 可面對中後段挑戰或 Boss
- 可達成明確通關或結局判定
- 可看到完整結果頁或結算回饋
- 可再次開局或回到主流程
- 遊玩過程中大量沿用現有視覺、世界、角色與 UI 資產
- 整體看起來必須像這個專案的落地版本，而不是臨時拼裝的陌生遊戲

你必須先讀的文件:
- `doc/文件索引.md`
- `doc/遊戲總覽/遊戲設計文件.md`
- `doc/遊戲總覽/開發進度清單.md`

- `doc/介面流程/玩家介面_按鈕與操作總清單.md`
- `doc/介面流程/玩家介面_戰鬥HUD與操作骨架.md`
- `doc/介面流程/主流程與頁面骨架_統一版.md`
- `doc/世界觀/世界觀與視覺圖鑑.md`
- `doc/世界觀/世界故事_輪迴敘事與四隊長人物設計.md`
- `doc/角色與敵方/角色圖鑑.md`
- `doc/角色與敵方/成員圖鑑.md`
- `doc/角色與敵方/怪物圖鑑.md`

- `doc/素材與提示詞/音樂與音效設計總表.md`

- `assets/images/maps/`

- `src/main.ts`

工作方式:
第一階段：全面審計
- 先從文件抽出真正的遊戲核心 loop、世界架構、角色定位、通關形式、介面流程
- 再從程式碼判斷哪些系統已存在、哪些已半完成、哪些只差接線、哪些真的缺
- 再從 assets 判斷哪些視覺、角色、世界、敵人、設施、音樂、影片能直接上線
- 做出一份你自己的內部判斷：哪些必須保留、哪些必須整合、哪些必須補完、哪些可以放棄

第二階段：定義完整成品的最小必要閉環
你不能只做 demo。你要先定義並完成至少一條真正可通關的遊戲閉環。至少包含:
- 主畫面 / 入口
- 開局或進場
- 隊伍/角色/配置承接
- 正式戰場
- 敵人與壓力成長
- 資源或成長回饋
- 中段推進
- Boss 或等價最終挑戰
- 勝敗條件
- 結算與回到主流程

第三階段：極限整合現有成果
你要把 repo 內既有成果盡量吃乾抹淨:
- 已有角色圖、頭像、透明立繪、icon，就接進 UI/HUD/圖鑑/隊伍顯示
- 已有世界地板、地圖素材、折皺層、物件圖、設施圖，就接進場景
- 已有世界影片、標題音樂、戰鬥音樂，就接進主畫面與遊玩流程
- 已有敵人圖、資料表、Boss、掉落、鍛造、召喚、寶箱、成長、死亡遺落、正式/訓練狀態，就盡可能接成真系統
- 已有頁面骨架、管理介面、主畫面、操作頁、結算頁，就不要廢掉，應改造成正式流程的一部分
- 已有資料模型、狀態管理、事件流，就盡量沿用，不要任性另開一套平行宇宙

強制規則:
1. 有現成 asset 就用現成 asset。
2. 有現成系統就接現成系統。
3. 有現成流程就補成真流程。
4. 只有在真的缺少關鍵功能時，才新增最小必要內容。
5. 不要把專案做成 showcase、沙盒、只有測試價值的工具。
6. 不要把正式流程讓位給 debug UI。
7. 不要讓訓練道場取代正式主線；它可以保留，但不能取代通關流程。
8. 你要優先做「正式可玩版本」，不是「驗收控制台更豪華」。
9. 你可以刪減野心，但不可以刪掉閉環。
10. 若完整四大世界來不及全部高完成度落地，至少要選一條最有條件的正式路線做到真正完整通關，並把其他內容最大化保留在可延展狀態，而不是半吊子平均鋪開。

決策優先級:
1. 可完整通關
2. 可穩定執行
3. 最大化沿用舊成果
4. 核心玩法清楚
5. 視覺與世界觀一致
6. 系統深度
7. 額外華麗功能

如果文件與程式碼不一致:
- 優先採用最接近「已能落地」的解釋
- 但不能背離整體世界觀與設計主軸
- 你要主動做收斂，而不是停在矛盾裡

如果內容太大做不完:
- 不准退化成 prototype
- 你必須縮成一條完整可玩的正式戰役 / 正式模式 / 正式世界線
- 但那條線一定要能通關
- 其他未完成內容保留骨架與資料相容性，方便後續擴充

實作要求:
- 修掉會阻止完整遊玩的斷點、假資料、占位流程、只展示不結算的頁面
- 接通主流程、正式戰場、管理頁、結算頁、再來一場
- 讓核心成長或配置至少有一部分真的影響戰鬥
- 讓敵人/Boss/掉落/資源/技能/成員/武器等至少形成一條真實互動鏈
- 優先把現有最成熟的內容整合成產品，而不是再發明新系統
- 可以重構，但只在有助於完整交付時進行
- 每次重大進展都做 commit

git 要求:
- 每完成一個里程碑都 commit
- commit message 必須清楚描述實際完成的內容
- 若環境允許，請推送遠端
- 如果 push 失敗，要保留本地 commit 並在最終報告中說明阻塞原因

里程碑建議:
- `audit: map existing systems, docs, and reusable assets for full rebuild`
- `feat: connect main menu, gameplay flow, and result loop into shippable route`
- `feat: integrate real combat progression and boss clear condition`
- `feat: reuse existing art, audio, and UI assets across the full playable build`
- `fix: close blockers preventing full clear and repeatable game sessions`

最終交付標準:
你不能用「還是 prototype」來描述最後成果。
你必須交付的是一個你認為已經達到「第一個正式可玩成品版」的版本，哪怕範圍收斂過，也必須是完整的。

最終你必須輸出:
1. 這次實際完成的完整遊玩路線
2. 玩家如何從開始一路玩到通關
3. 你沿用了哪些既有文件決策
4. 你沿用了哪些既有程式碼模組
5. 你沿用了哪些既有 asset
6. 你新增了哪些必要內容
7. 還有哪些缺點存在，但為什麼它仍然算是可交付成品
8. 如何啟動、驗收、重玩
9. 所有 commit 與 push 結果

最後提醒:
- 你不是來做參考稿。
- 你不是來做漂亮的半成品。
- 你不是來做說服人的簡報。
- 你是來把這個 repo 內已存在的一切可用成果，壓榨、整合、收斂成一個真的能玩到結束的版本。
- 能用舊東西就用到極致。
- 能接起來就不要重做。
- 有路就把它鋪到能通關。
- 除非真的被硬性阻塞，否則不要停。
```
